# 《测试计划》白盒测试部分

## 1. 测试目标

本测试计划面向 `RAG-Knowledge-Assistant` 项目的练习管理模块与课程社区模块，针对成员 C 负责的白盒测试内容进行设计。测试目标如下：

1. 验证 `ExerciseController`、`PostController` 及其关联 `service`、`repository` 层在正常输入、异常输入和边界输入下的控制流与数据流是否符合预期。
2. 通过自动化测试覆盖练习题查看、习题集维护、帖子浏览、发帖、评论、回复等关键处理逻辑，发现分支遗漏、异常处理不完整、权限控制失效和数据映射错误等问题。
3. 采用代码覆盖、基本路径覆盖与数据流分析相结合的方法，提高本人负责模块的语句覆盖、分支覆盖和关键 def-use 路径覆盖水平，并为项目整体覆盖率达到 95% 以上提供支撑。
4. 为后续《测试报告》中的缺陷分析、覆盖率统计和未来扩展适配提供可追踪的测试设计依据。

5. 汇总测试执行结果、覆盖率结果与发现的问题，形成《测试报告》。

## 5. 测试方法

本计划采用代码覆盖、基本路径覆盖和数据流分析三类白盒测试方法。

### 5.1 代码覆盖法

目标是覆盖本人负责模块中的主要语句、判断和异常处理路径，重点关注：

1. 练习模块中的成功分支与异常分支
   - 上传成功与上传异常
   - 详情查询成功与未命中异常
   - 删除成功与删除失败
   - 更新习题时教师/非教师权限分支
   - 更新目标存在/不存在分支
2. 社区模块中的主要控制分支
   - 帖子列表查询关键字为空/非空分支
   - 匿名/实名帖子作者映射分支
   - 评论内容为空/非空分支
   - 评论和回复的 `parent_id` 为空/非空分支
   - 帖子详情存在/不存在分支
   - 匿名/实名评论展示分支

计划优先覆盖控制器中的所有显式判断语句，以及仓储层中分页、过滤、保存与删除相关代码。

### 5.2 基本路径覆盖法

将以控制器与仓储层中具有判定节点的方法为核心，识别独立路径并设计至少一条测试用例覆盖每条基本路径。重点路径包括：

1. `update_exercise()` 的权限拒绝路径、更新成功路径、更新失败路径。
2. `get_exercise_set_detail()` 的查询成功路径与查询失败路径。
3. `delete_exercise_set()` 的删除成功路径与删除失败路径。
4. `search_exercise_sets_with_pagination()` 的无关键字路径与有关键字路径。
5. `get_course_posts()` 的无关键字分页路径与关键字过滤路径。
6. `add_comment()` 的空评论拒绝路径、普通评论成功路径、回复评论成功路径、匿名评论成功路径。
7. `get_post_by_id_api()` 的存在路径与 404 路径。

### 5.3 数据流分析法

将围绕关键变量的定义、使用、传递与输出设计测试，重点分析以下数据流：

1. 练习题集创建数据流
   - `ExerciseSetCreateDTO` 中 `course_id/title/description/document_id/document_name/questions`
   - `exercise_set.id` 在保存习题集后被定义，并流向每道 `Exercise.exercise_set_id`
   - `q.options`、`q.answer` 从 Python 对象转换为 JSON 字符串并写入数据库
2. 练习题查询与更新数据流
   - `keyword` 从控制器传入仓储层，决定查询过滤条件和分页结果
   - `set_id`、`exercise_id` 影响记录检索、异常抛出与最终响应
   - `current_user.role` 影响更新逻辑是否进入 service/repository
3. 社区帖子数据流
   - `keyword.strip()` 影响帖子列表过滤逻辑
   - `user.id/user.username/user.role` 与 `is_anonymous` 共同决定响应中 `author`、`author_role`
4. 评论与回复数据流
   - `request.content` 先经过 `strip()` 判空，再流入评论创建逻辑
   - `parent_id` 决定评论是一级评论还是回复
   - `created_at` 在评论创建时定义，并在返回 DTO 时格式化输出

对于上述数据流，测试将重点验证“定义后是否被正确使用”“分支切换后输出是否保持一致”“异常分支是否导致数据流中断并返回正确错误信息”。

4. 结果整理阶段

## 8. 风险与说明

1. 当前项目中“练习题作答反馈”主要由前端页面本地实现，后端没有独立的“提交答案/判题”接口，因此白盒测试计划中对该需求的后端覆盖有限，需要在报告中明确这一实现边界。
2. 部分 service 层仅作简单转发，实际分支主要集中在 controller 与 repository 层，因此覆盖率统计应按真实代码复杂度解释，而不能只看模块名称。
3. 当前评论新增逻辑仅校验“评论内容不能为空”，尚未显式校验 `post_id` 是否存在、`parent_id` 是否有效；若测试中发现可插入非法关联数据，应作为缺陷记录而非跳过。
4. `ExerciseController.update_exercise()` 仅允许教师更新，这是代码中的访问控制要求；若需求后续要求学生可提交作答记录，则该部分需要重新设计并补充新的白盒测试路径。
5. 仓储层大量依赖数据库状态，若测试环境数据不隔离，可能导致分页数量、查询结果和覆盖率统计不稳定，因此执行时必须使用隔离测试库或事务回滚策略。
6. 若后续项目新增“习题作答记录持久化”“帖子删除/编辑”“评论树形展开”等功能，本测试计划需同步扩展控制流、数据流和覆盖目标。
