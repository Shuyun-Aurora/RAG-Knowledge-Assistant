# 未来扩展适配方案

## 1. 说明

本部分针对成员 C 负责的练习管理模块与课程社区模块，结合当前系统实现情况，推测未来可能出现的功能变更，并给出对应的黑盒测试与白盒测试调整方案。

选择这些变更的依据如下：

1. 当前练习模块后端主要支持习题集管理，尚未提供作答记录持久化与判题接口，存在明显的扩展空间。
2. 当前课程社区模块已支持发帖、评论、回复，但缺少内容治理与审核能力，随着用户量增大通常需要新增管理功能。
3. 当前评论结构较基础，未来若社区互动增强，常见演进方向是支持更完整的楼中楼、提醒或引用机制。

## 2. 未来变更一：新增“练习作答记录与成绩统计”功能

### 2.1 变更描述

当前系统中，练习题作答与反馈主要由前端页面本地完成，后端没有独立的“提交作答”“保存作答记录”“查询历史成绩”接口。未来系统可能新增如下功能：

1. 学生提交练习答案后，后端保存每次作答记录。
2. 后端自动判题并生成分数、正确率、错题列表。
3. 学生可查看自己的历史作答记录。
4. 教师可查看某习题集的学生完成情况统计。

### 2.2 对黑盒测试的调整方案

若该功能上线，黑盒测试需要新增以下测试点：

1. 学生提交完整答案时，系统应返回分数、正确题数、错误题数和反馈信息。
2. 学生重复提交同一习题集时，应根据需求校验是覆盖旧记录、保留多次记录还是仅允许首次提交。
3. 学生提交空答案、部分答案、非法题号答案时，系统应返回明确错误提示。
4. 学生查询个人作答历史时，应只看到自己的记录。
5. 教师查询完成情况时，应能看到课程内学生的完成率与成绩统计。
6. 未登录用户或无权限用户访问作答记录接口时，应被拒绝。

### 2.3 对白盒测试的调整方案

若新增后端作答逻辑，白盒测试需要新增或扩展以下内容：

1. 新增作答提交控制器的语句覆盖与分支覆盖
   - 正常提交路径
   - 重复提交路径
   - 参数校验失败路径
   - 权限拒绝路径
2. 新增判题服务层的基本路径覆盖
   - 单选题正确/错误路径
   - 多选题完全正确/部分正确/完全错误路径
   - 填空题命中/未命中路径
3. 新增成绩统计逻辑的数据流分析
   - 学生答案从请求 DTO 到判题结果对象再到数据库记录的流转
   - `score`、`correct_count`、`wrong_count` 等统计变量的定义与使用路径
4. 新增作答记录仓储层测试
   - 新记录插入路径
   - 历史记录查询路径
   - 按学生、按习题集过滤查询路径
5. 覆盖目标调整
   - 新增作答相关 `controller/service/repository` 文件后，应将其纳入练习模块总体覆盖率统计
   - 重点保证判题服务和成绩汇总逻辑的分支覆盖率不低于 95%

### 2.4 风险说明

该变更会显著提升练习模块代码复杂度，尤其是多题型判题和历史记录查询逻辑。测试中应重点关注边界输入、判题正确性、重复提交规则和权限控制是否一致。

## 3. 未来变更二：新增“社区管理员/教师审核与删帖管理”功能

### 3.1 变更描述

当前社区模块支持发帖、评论和回复，但没有内容审核、删除、屏蔽或举报处理能力。未来系统可能新增如下功能：

1. 新增社区管理员角色，或赋予教师额外的社区管理权限。
2. 支持帖子删除、评论删除、内容屏蔽、违规内容审核。
3. 支持用户举报帖子或评论。
4. 被删除内容在前端不再正常展示，或仅保留“内容已删除”占位。

### 3.2 对黑盒测试的调整方案

若该功能上线，黑盒测试需要新增以下测试点：

1. 普通学生尝试删除他人帖子或评论时，应被拒绝。
2. 教师或管理员删除帖子、评论后，列表与详情页展示应符合需求。
3. 被删除帖子是否仍允许查看历史评论，需按需求设计对应测试。
4. 举报提交成功后，系统应记录举报信息并返回明确提示。
5. 对已删除或已屏蔽的帖子继续评论、回复时，应被拒绝或按需求限制。
6. 管理员审核违规内容后，处理状态应正确更新。

### 3.3 对白盒测试的调整方案

若新增社区管理逻辑，白盒测试需要新增或扩展以下内容：

1. 新增权限控制分支覆盖
   - 学生无权删除路径
   - 教师可删除路径
   - 管理员可删除/审核路径
2. 新增帖子与评论删除逻辑的基本路径覆盖
   - 目标存在/不存在路径
   - 软删除/硬删除路径
   - 已删除对象重复删除路径
3. 新增举报与审核服务层的数据流分析
   - 举报请求中的 `target_id`、`target_type`、`reason` 的传递路径
   - 审核状态从“待处理”到“已处理”的状态流转
4. 调整现有帖子列表与评论列表白盒测试
   - 需要新增“过滤已删除内容”的查询路径
   - 需要验证删除标记字段对 DTO 映射和最终响应的影响
5. 调整覆盖率统计范围
   - 若新增 `ModerationController`、`ReportService`、`ReportRepository` 等文件，应纳入社区模块覆盖率结果

### 3.4 风险说明

该变更的核心风险是权限控制错误和删除后数据不一致。测试中应重点关注不同角色访问控制、删除后查询行为和审核状态变更是否正确。

## 4. 未来变更三：增强评论功能，支持多级回复与@提醒

### 4.1 变更描述

当前评论功能已支持通过 `parent_id` 实现基础回复，但结构较简单。未来社区模块可能进一步增强为：

1. 支持多级楼中楼评论。
2. 支持 `@用户` 提醒。
3. 支持引用回复或展示评论树。
4. 支持按照“最新回复”“最热评论”等方式排序。

### 4.2 对黑盒测试的调整方案

若该功能上线，黑盒测试需要新增以下测试点：

1. 用户回复一级评论、二级评论、三级评论时，系统应正确展示层级结构。
2. 用户输入 `@用户名` 后，系统应正确识别提醒对象。
3. 回复已删除评论或被屏蔽评论时，应按需求拒绝或特殊处理。
4. 评论树展开、折叠和排序切换时，前端展示应与后端返回结构一致。
5. 提醒对象不存在、重复提醒、提醒自己等特殊场景应有明确定义。

### 4.3 对白盒测试的调整方案

若该功能上线，白盒测试需要新增或扩展以下内容：

1. 扩展评论创建逻辑的分支覆盖
   - `parent_id` 为空路径
   - `parent_id` 指向一级评论路径
   - `parent_id` 指向更深层评论路径
   - 层级超限或非法层级路径
2. 新增评论树构造逻辑的基本路径覆盖
   - 无子评论路径
   - 单层子评论路径
   - 多层递归组装路径
3. 新增提醒逻辑的数据流分析
   - `mention_user_id` 或解析出的提醒对象从请求文本到消息/通知记录的传递路径
   - 评论创建与提醒创建之间的事务一致性
4. 调整评论查询测试
   - 从当前的线性列表断言扩展为树形结构断言
   - 增加排序字段对查询结果顺序影响的测试
5. 覆盖率调整建议
   - 若评论树组装、提醒通知等逻辑转入独立服务类，应将其单独纳入分支覆盖重点

### 4.4 风险说明

该变更的复杂点主要在于递归评论树组装、层级限制和提醒对象解析。测试中应重点关注深层回复、非法父节点、树结构顺序和提醒对象准确性。

## 5. 总体适配结论

结合当前项目实现情况，练习管理模块与课程社区模块未来最可能的演进方向包括：

1. 练习模块从“静态查看与本地反馈”扩展为“后端持久化作答与成绩分析”。
2. 社区模块从“基础互动”扩展为“带治理能力的课程社区”。
3. 评论功能从“简单回复”扩展为“多级互动与提醒机制”。

从测试适配角度看，未来功能变更后需要同步调整两类测试：

1. 黑盒测试需要新增业务规则验证、权限验证、异常输入验证和前后端交互结果验证。
2. 白盒测试需要新增控制器、服务层、仓储层的语句覆盖、分支覆盖、基本路径覆盖和关键数据流分析。

因此，在项目后续迭代中，建议继续保持“先识别需求变更点，再同步更新测试计划、测试用例、测试代码和覆盖率分析”的测试工作流，以确保未来功能扩展后系统仍具备较高的可测性与稳定性。
